# 今晚对话总结 - 高英华 / lian-codex

时间：2026-07-12

## 范围

这份只总结今晚高英华和 lian-codex 讨论的 agent 轨迹字段与采集机制，不总结张耀明裁决线，也不把聊天纠错筛选流程当主线。

## 1. 先确认的三类 agent 轨迹采集方式

今晚讨论的不是单一 JSONL schema，而是三种采集方式混在同一个脱敏样例文件里：

1. **Claude Code / claudecode：wire 透传旁路**
   - 通过 Anthropic API 透传代理抓 `/v1/messages` 请求和响应。
   - 原始请求里有外层 `system`、`tools`、`messages`、`model`。
   - `system` 和 `tools` 只存在于发往 API 的请求体里，本地 transcript 和 OTel 不一定拿得到，所以需要 wire 旁路。
   - tool result 在 Anthropic 协议里常见为 `role:"user"` 消息里的 `type:"tool_result"` block，而不是 OpenAI Chat Completions 那种顶层 `role:"tool"`。

2. **lian-codex / Codex：wire 代理逼 WebSocket 退 HTTP**
   - 通过 Codex capture proxy 抓真实 `/responses` 调用。
   - 请求里核心字段是 `instructions`、`input`、`tools`、`model`、`reasoning` 等。
   - adapter 会把 `request.instructions` 放进归一化 `messages` 的首条 `role:"system"`，`request.input` 和 `final_response.output` 转成 OpenAI messages 形态。
   - Codex 的 tool schema、usage、sampling 回显比较干净；但 reasoning 是加密块，明文拿不到，只能剥掉并记录 `n_reasoning_stripped`。

3. **Antigravity / zym-antigravity：离线读产品自落 transcript**
   - 读 Antigravity 自己落盘的 `transcript.jsonl`。
   - 这不是 wire 级请求，只有会话级事件流。
   - system prompt 可能被 bridge 内联进 `<USER_REQUEST>` 或 `CONVERSATION_HISTORY`，adapter 只能从这些文本块里切出来。
   - tool schema、usage、采样参数不是官方 wire 原件，缺失或需要反推，不能冒充全字段。

## 2. JSONL 里 `role=user` 包 tool 返回不是异常

你指出“中间很多明明是 tool 返回结果，但 role 是 user”。这个现象不能按 OpenAI Chat Completions 的 `{"role":"tool"}` 直觉判断。

对 Claude/Anthropic 风格，工具返回通常是 user 消息里的 content block：

```json
{
  "role": "user",
  "content": [
    {"type": "tool_result", "tool_use_id": "...", "content": "..."}
  ]
}
```

语义上它是工具观察结果，但协议角色上挂在 user turn 里。后续如果转成我们自己的归一化 SFT / replay 格式，可以再映射成 `role:"tool"`；但原始 wire 里看到 `role:"user"` 不代表它是人说的话。

## 3. `messages` 里的 system 和外层 system 不是同一个东西

今晚反复核对的关键点是：不能只看字段名叫 `system` 就认为来源相同。

在 Claude wire 样例里：

- 外层 `request.system` 是 Anthropic API 请求体的系统提示，可能是 block list，里面包含 Claude Code 身份、运行规则、billing header 等。
- `request.messages` 是发给模型的对话消息列表；其中可能有 `<system-reminder>`、工具返回、历史上下文等内容。
- 这些 `messages` 里的系统样文本不等于外层 `system`，它们通常是 agent harness 自己塞进对话历史的 reminder/context。

在 Codex wire 归一化里：

- 原始外层是 `request.instructions`，不是 `system` 字段。
- adapter 为了统一训练/回放格式，会把 `instructions` 转成 `messages[0] = {"role":"system", ...}`。
- 所以这里 `messages` 内的 system 是 adapter 从原始请求字段转出来的，原始出处是 `request.instructions`。

在 Antigravity transcript 里：

- 没有官方 wire 外层 system。
- adapter 只能从 `<USER_REQUEST>`、`CONVERSATION_HISTORY`、`SYSTEM_MESSAGE` 等产品 transcript 文本里切出 system-like 消息。
- 这类 system 是离线重建结果，不是闭源服务端最顶层官方 system。

## 4. “归一化”在这里有体现，但不能乱套到所有字段

归一化的目的，是把三家不同协议压成一套可训练、可回放、可评测的共同表示，例如统一成 `messages`、`tools`、`usage`、`sampling`、`fidelity`、`completion_start_index` 等字段。

但今天的纠正点是：不能看到 `messages` 里有 `system`，就直接说“这是归一化加的 subtype”。必须看具体采集方式和原始出处：

- Claude wire 原始请求本身就有外层 `system` 和 `messages`，不是 adapter 随便造的。
- Codex adapter 确实会把 `instructions` 归一化为 `messages` 内 system。
- Antigravity adapter 会从产品 transcript 中切 system-like 内容，这是离线重建。

所以正确说法是：归一化存在，但每个 system-like 字段都要追到原始采集源，不能只按目标 schema 解释。

## 5. 从 tokenization 角度，谁决定内外 system 怎么拼

今天讨论到 Qwen / GLM / 开源模型时，结论要拆层：

1. **网关或 agent harness 决定送进去哪些字段**
   - 它可能保留外层 system，也可能把外层 system 拼进 messages，或把工具列表塞成一条 system/reminder。
   - 这一层如果能看代码或抓 wire，就可审计。

2. **chat template 决定 messages 如何被串成 token 序列**
   - 对开源模型，比如 Qwen/GLM，如果最终调用 Transformers tokenizer 的 `apply_chat_template`，那 messages 里的 `role:"system"` 会按模板被渲染成特殊标记和文本。
   - 但“外层 system”如果不是 messages 的一部分，chat template 本身看不到它；它必须先被网关/harness 拼进 messages 或 prompt。

3. **模型训练后的角色遵循方式决定冲突时听谁**
   - 即使 token 序列能看清，冲突时模型更听外层还是内层，还受训练和服务端策略影响。

因此，“内外 system 怎么拼”不是单由模型名决定。可审计链路是：agent harness / 网关先把外层字段变成上游请求，上游服务端或本地 tokenizer 再按 chat template 串成最终 token。

## 6. 轨迹边界怎么定

最后讨论的问题是：群里随时有人问问题，哪些对话算同一条 agent 轨迹。

当前可以讲清楚的机制分三层：

1. **在线路由层**
   - 按回复链、上下文、当前目标，把群消息路由到某个会话线。

2. **离线任务重切层**
   - 不按“谁在说话”简单切，而是按同一个目标或交付物归并。

3. **轨迹挂接层**
   - 把对话线和真实 agent 调用轨迹关联起来。
   - Claude wire 路可以用 `session_id`、`thread_id`、`system+tools` 指纹区分主线、子代理、压缩变体。
   - Codex wire 路每个 `/responses` call 有 `completion_start_index`，天然适合切 input/context 和 completion。
   - Antigravity 是会话级 transcript，需要靠压缩段、conversation history 和 per-turn context 重建近似输入。

当前口径：同一条轨迹应围绕同一个任务目标/交付物，而不是简单按时间窗口、发言人或业务标签切。

## 7. 我前面写错的地方

我前面写成“聊天记录纠错筛选流程、gold、t0/B/C、重新筛选”等内容，和你今晚问的 agent 轨迹字段细节不对应。那份总结不应作为今晚你我对话的总结。

正确主线应是：

- 三种采集方式分别能抓到哪些字段；
- 为什么 tool 返回会在 `role=user` 里；
- `messages` 内 system 和外层 system 的区别；
- 归一化到底有没有发生、发生在哪；
- tokenization / chat template / harness / 网关分别决定什么；
- 群聊场景下怎样判断哪些消息属于同一条 agent 轨迹。
