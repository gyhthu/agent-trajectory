# 三家 Agent 轨迹劫持机制（claudecode / antigravity / codex）

> 版本：2026-06-25　起草：lian-server(claude)　数据负责人：张耀明
> 写法参照：飞书《slime coding_agent_rl 的 API 劫持机制分析》
> 关联：`2026-06-24-三家agent轨迹劫持保真度总账.md`、`训练数据归集方案.md`

## 一句话定位

让三家**未经修改**的真实 agent（claudecode=我自己 / zym-antigravity / lian-codex）照常在群里干活，通过**在请求线上截获**或**离线读产品落盘**，把每一次模型交互的完整轨迹（system / tools / 多轮 messages / usage / reasoning）原样记录下来，产出可直接做 SFT / BC 的训练样本。**Agent 框架零侵入，回复内容零改动。**

**和 slime 的本质区别**——这点先说清，免得混淆：

| | slime coding_agent_rl | 我们（本文） |
|---|---|---|
| 目的 | **RL rollout**：在线训练 | **离线 SFT/BC**：采真实生产轨迹 |
| shim 干什么 | **改道**：伪装成 API，把流量劫到**自家训练模型** | **透传**：原样转发真实上游，只**旁路记录**不改一字 |
| agent 跑的是 | 训练中的模型（slime-actor） | **真实生产模型**（线上 bot 照常服务） |
| 要存什么 | token id + logprob（token 级，RL 必须） | **逐字段原文**（string 级保真，BC/SFT 够用） |
| 评分 | 双沙箱跑测试出 reward | 无（不是 RL，不需要 reward） |

一句话：slime 是「**把 agent 骗去喂训练模型**」，我们是「**在 agent 真干活时旁边录一份完整账**」。同源思路（换 BASE_URL 这个劫持点），不同目的。

## 总体架构

```
                        三家 agent 照常在飞书群干活
   ┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐
   │  claudecode (我)     │  │  antigravity        │  │  codex (lian-codex) │
   │  ANTHROPIC_BASE_URL  │  │  闭源 Go LS         │  │  ChatGPT OAuth      │
   │   → 127.0.0.1:4319   │  │  gRPC/TLS+证书pin   │  │  默认 wss://        │
   └──────────┬──────────┘  └──────────┬──────────┘  └──────────┬──────────┘
   ① wire透传截获          ③ wire拦不到→离线读      ② 逼WS退HTTP→wire截获
              │                        │                        │
              ▼                        ▼                        ▼
   ┌─────────────────────┐  ┌─────────────────────┐  ┌─────────────────────┐
   │ anthropic_capture_   │  │ 产品自落 transcript │  │ codex_capture_      │
   │  proxy.py :4319      │  │ ~/.gemini/.../      │  │  proxy.py :4320     │
   │ 原样转发真实上游     │  │   transcript.jsonl  │  │ proxycap provider   │
   │ + 双轨落盘           │  │ （事后成品账本）    │  │ supports_ws=false   │
   └──────────┬──────────┘  └──────────┬──────────┘  └──────────┬──────────┘
              │ api-calls/*.jsonl.gz   │ 会话事件流             │ codex-api-calls/*.gz
              ▼                        ▼                        ▼
   ┌───────────────────────────────────────────────────────────────────────┐
   │  归一化 adapter（trajectory_aggregate / *_adapter.py）                  │
   │  → 同构轨迹 JSONL → secret-scrub 脱敏 → canonical IR(OpenAI-messages)   │
   │  → SFT / 行为克隆样本                                                    │
   └───────────────────────────────────────────────────────────────────────┘
```

劫持点离 wire（网络字节层）越近 → 保真度越高。**这正是三家成色差异的第一根因**（第二根因是服务端肯不肯下发明文 reasoning，见后）。

## 一、claudecode（我自己）：wire 透传截获，最高保真

### 劫持入口：只改一个环境变量
claude CLI 设 `ANTHROPIC_BASE_URL` 指到本地透传代理即接入：

```
ANTHROPIC_BASE_URL = http://127.0.0.1:4319    # 默认 127.0.0.1，安全规则绝不绑 0.0.0.0
```

agent 完全无感——CLI 原样跑，凭证（Authorization/OAuth token）照常发往真实上游，**代理不落盘任何请求头**（token 不碰磁盘）。

### 透传姿势：原样转发，不改一字
代理不是改道，而是**透明转发**：所有请求原样转给 `https://api.anthropic.com`，响应流式回传给 CLI。捕获只是**旁路**——出错只打日志不阻断转发（`观测不能搞挂生产 bot`），转发本身失败才返回 502。所以**线上回复质量零影响**。

### 核心契约：API 调用 = 训练样本（借 slime 的关键洞察）
`system` / `tools` 只存在于**发往 API 的请求体**里，模型 on-wire 原文只存在于**响应**里——本地 transcript 和内置 OTel 都拿不全，**在请求线上截获是唯一拿法**。

双轨落盘：
- `requests/YYYY-MM-DD.jsonl`：system+tools 去重轻量轨（聚合器 join 用）；
- `api-calls/YYYY-MM-DD.jsonl.gz`：**全量轨**——完整 request payload（带全上下文前缀，故 gzip）+ response（SSE 流重建成最终 message，含 thinking / tool_use / usage）。

### 多 agent 身份天然区分
主线 agent / 子代理(Task) / compaction / haiku 工具调用**全走同一个 `/v1/messages` 端点**，天然全覆盖；按 `sha(system+tools)` 聚类即可把它们分成不同 agent 身份。session 从 `metadata.user_id` 解析（新版 JSON 串 / 旧版 `session_*` 都认）。

> **拿到什么**：逐字节输入边界 ✅、system 原文 ✅、tool schema ✅、**原始 thinking 逐字真文本 ✅**（三家里唯一）、usage ✅（但 cache 命中时 `input_tokens` 会骗人）、采样参数 ✅。
> **脚本正本**：`trajectory-collect/scripts/anthropic_capture_proxy.py`

## 二、antigravity：wire 拦不到，离线读产品账本

### 为什么不能 wire 截获
antigravity 出口是**闭源 Go `language_server`**，走 gRPC/TLS + **证书 pinning**——MITM 代理拦不到（pinning 会拒掉自签证书）。`ANTHROPIC_BASE_URL` 那套对它无效。

### 劫持姿势：离线读它自己落的 transcript
退而求其次——**读产品事后写的"成品账本"**：

```
~/.gemini/antigravity/brain/<conv>/.../transcript.jsonl
```

这不是 wire 逐字节，而是**会话级重建**：产品把一次会话整理成事件流落盘，我离线读它。所以是「1 个 transcript = 1 会话事件流 = 1 条会话级记录」（`per_call_boundary=False`）。

### 代价：几项要反推或缺失
因为没碰 wire，几个只存在于请求体/响应原文的字段拿不到：
- **tool schema**：只有调用（name+args）没有定义 → 反推补；
- **采样参数 / usage**：产品账本里没有 → 缺失，adapter 里标 `fidelity=False` 不冒充；
- **system**：埋在摘要里，要重解析；
- **reasoning**：是产品**整理稿**——删改过、但**可读**（比 codex 强）。

> **拿到什么**：用户/assistant 文本 ✅、工具调用+结果 ✅、reasoning 🟡（二手可读稿）；tool schema/usage/采样参数 ❌（反推或缺）。
> **脚本正本**：`trajectory-collect/scripts/antigravity_transcript_adapter.py`（main `feb46a8`）；脱敏 SFT 集 `/opt/shared/data/antigravity-trajectory/`（权限 600）

## 三、codex（lian-codex）：逼 WS 退 HTTP，把 wire 抢回来

### 难点：默认走 WebSocket，HTTP 代理抓不到
codex 走 ChatGPT OAuth（`auth_mode=chatgpt`），主模型默认 **WebSocket(`wss://chatgpt.com`)**——它**无视 base_url**，HTTP 透传代理根本截不到流量。

### 破法：自定义 provider 逼它退回 HTTP
```
model_provider     = proxycap
supports_websockets = false       # ← 关键：逼 codex 从 WS 退回 HTTP
CODEX_CAPTURE_PROXY_URL = http://127.0.0.1:4320
```
codex 被迫改用 **HTTP POST `/backend-api/codex/responses`**，再经代理透传 → 抓到真实 wire 字节（请求体 zstd 压缩，代理解压）。这样 wire 又回到手里了。

兜底：另有 `~/.codex/sessions/*.jsonl` 自落盘 rollout，wire 没开时也有粗轨。

### 透传姿势 & 线上影响
和 claude 一样是**透传不改道**：只改传输通道（WS→HTTP），`/responses` 逻辑载荷逐字段一致 → **回复质量零影响、登录身份零影响、存量数据零影响**。唯一体感差异是流式平滑度（SSE vs WS）可能轻微。opt-in 默认关，回滚=删 env 重启，瞬间回原生 WS 直连。

### 意外收获 & 唯一死穴
- ✅ **格式占便宜**：codex 原生就是 OpenAI `/responses`，离 canonical IR 最近；tool schema 19 条全字段、usage 最干净（cached/reasoning_output 分列）；采样参数在**响应** `final_response` 里回显（`temp=1.0/top_p=0.98/effort=medium`）。
- ❌ **reasoning 永久加密**：wire 响应里 reasoning item 是 `summary=[] + content=[] + 1592 字节 encrypted_content`，整条 SSE 流 grep `reasoning_*_text` 事件 = **0 次**。根因是 **OpenAI 服务端对 ChatGPT 客户端就不下发明文 reasoning**，wire 级也救不回——**不是采集姿势问题，是服务端策略**。

> **拿到什么**：逐字节输入边界 ✅、system 全文 ✅、tool schema 19 条 ✅、usage 最干净 ✅、采样参数 ✅（响应侧）；**reasoning ❌ 加密锁死**。
> **脚本正本**：`codex_capture_proxy.py`（`5b22917`）+ `codex_wire_adapter.py`（`c722dd4`）

## 五、结论：三家成色与训练分工

| Agent | 成色 | 能拿去干嘛 |
|---|---|---|
| **claudecode** | 全能源 | **唯一**能做原始推理监督(CoT SFT)；BC 也全字段 |
| **antigravity** | 残但可用 | BC 够用；reasoning 二手稿（可读，比 codex 强）；缺 tool schema/usage 反推补 |
| **codex** | BC 最佳、推理出局 | 除 reasoning 全字段齐、格式最贴 IR、usage 最干净 → **BC 首选**；CoT 没戏 |

**最干净的切法**：
- **行为克隆(BC)**：三家都喂，**codex/claude 为主**（格式最顺）；
- **带 CoT 的推理监督**：**只切 claudecode 那片**（三家里唯一有真思考文本）。

保真度差异的两条根因，到这里收口：
1. **劫持点离 wire 多近** → claude/codex 在 wire（全字段），antigravity 离线重建（缺字段）；
2. **服务端肯不肯下发明文 reasoning** → claude 给（真品），antigravity 给整理稿，codex 永久加密（wire 也救不回）。

## 六、对我们的启发（接 slime 那篇）

1. **零侵入轨迹收集 = 在代理层做，不在 agent 里埋点**。换 BASE_URL / 逼退 HTTP 这个劫持点，对任何 SDK 兼容的 agent 通用——claudecode、codex 都验证了；antigravity 因证书 pinning 走不通，是这条路的边界。
2. **采集与训练彻底解耦**：透传代理只是采集期旁路组件，不进训练管线、不进最终模型。代理关掉/回滚不影响已收数据，训出的模型对代理/上游零运行时依赖。
3. **混源要先统一输入格式**：codex(上下文无 CoT) 和 claude(带 thinking) 混进同一 SFT 集会分布不一致 → 要么统一抹平(全集纯 BC)，要么显式标记(学"何时该想")。
4. **拼不回来的就 mask/剥，不硬训**（同 slime 的 provenance 守卫）：codex 多轮夹的加密 reasoning 块训练前必剥，只留 `n_reasoning_stripped` 计数，不塞别的模型隐状态。

## 七、诚实边界（不冒充）
- codex reasoning 明文：服务端加密，wire 级也拿不到，**不伪造**，剥块记计数；
- antigravity 的 system/tools/usage：反推或缺失，adapter 里标 `fidelity=False`，不冒充官方原件；
- 本文所有字段结论均来自**实测**（wire 抓包解压 + rollout 字段统计），推断处已显式标注。
