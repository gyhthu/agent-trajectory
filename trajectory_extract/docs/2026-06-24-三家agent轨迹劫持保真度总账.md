# 三家 Agent 轨迹劫持保真度总账（claude / antigravity / codex）

> 版本：2026-06-24　起草：lian-server(claude)　数据负责人：张耀明
> 关联文档：`2026-06-23-antigravity轨迹SFT方案与交接.md`、`训练数据归集方案.md`

## 0. 一句话

把三家 agent（claude=我自己 / zym-antigravity / lian-codex）的工作轨迹劫持下来做 SFT，**最终保真度差异的根因只有两条**：
1. **劫持点离 wire（网络字节层）多近**；
2. **模型服务端肯不肯下发明文 reasoning（思考过程）**。

结论先行：**行为克隆（BC）三家都能喂，codex/claude 为主、格式最顺；要带 CoT 的推理监督，只能切 claude 那片。**

## 1. 三家是怎么被劫持的（采集姿势）

| Agent | 劫持方式 | 本质 | 落盘位置 |
|---|---|---|---|
| **claude（我）** | wire 劫持代理 `127.0.0.1:4319`（`ANTHROPIC_BASE_URL` 重定向） | 网络层逐字节拦真实 `/v1/messages` 请求+响应 | `trajectory-data/api-calls/` |
| **antigravity** | **离线读**产品自落的 transcript 会话日志 | 没碰 wire，读它事后写的"成品账本"（出口是闭源 Go `language_server`，gRPC/TLS+证书 pinning，MITM 拦不到） | `~/.gemini/antigravity/brain/<conv>/.../transcript.jsonl` |
| **codex** | **两条腿**：①wire 代理 `127.0.0.1:4320`（`CODEX_CAPTURE_PROXY_URL`，逼 WS→HTTP）✅2026-06-24 上线 ②自落盘 rollout 兜底 | 能到 wire，但服务端对 reasoning 上锁 | `trajectory-data/codex-api-calls/` & `~/.codex/sessions/*.jsonl` |

### codex wire 代理破解要点
- codex 走 ChatGPT OAuth（`auth_mode=chatgpt`），主模型默认 **WebSocket(`wss://chatgpt.com`)**，无视 base_url，HTTP 代理抓不到；
- 破法：自定义 `model_provider=proxycap` + `supports_websockets=false`，**逼 codex 从 WS 退回 HTTP POST `/backend-api/codex/responses`**，再经代理透传 → 抓到真实 wire 字节（请求体 zstd 压缩，代理解压）；
- 脚本正本：`trajectory-collect/scripts/codex_capture_proxy.py`（git `5b22917`），opt-in，默认关。

## 3. 最大差异：reasoning（三家天差地别）

| | 思考过程最终拿到什么 | 档位 |
|---|---|:---:|
| **claude** | 原始 thinking block **逐字真文本** | 🥇 真品 |
| **antigravity** | 产品**整理稿**——删改过、但**可读** | 🥈 二手可读 |
| **codex** | `summary=[]` + `content=[]` + 1592 字节**加密块**(`gAAAAAB…`)，流式全程零 reasoning 文本 | 🥉 锁死不可读 |

### codex reasoning 加密铁案（wire 级实测）
- 从 wire 响应里的 2 个 reasoning item 直接看：`summary=[]`、`content=[]`，唯一有料的 `encrypted_content` 是 1592 字节加密二进制，客户端解不开；
- 整条 SSE 原始流（36 万字节）grep：`response.output_text.delta` 74 次（正文逐字吐，**有**）；`reasoning_summary_text` 等任何 reasoning 文本事件 **= 0 次**；
- 根因：**OpenAI 服务端对 ChatGPT 客户端就不下发明文 reasoning**，那串密文是给"多轮把 reasoning 回传续上下文"用的、**故意对客户端不可读**。wire 级都救不回——**这不是采集姿势问题，是服务端策略。**

## 4. codex wire 代理上线：实测 & 线上影响 & 回滚

### 已实测（2026-06-24，连续运行验证）
- 代理进程连续跑 35 分钟，RSS 稳定无泄漏，bridge 全程 `active`，日志 error/retry/fail **0 条**；
- 联调那轮 `status=200` 正常回话，抓到 141KB wire 数据，含 19 条 tool schema 全字段。

### 对 lian-codex 线上使用的影响
| 维度 | 结论 |
|---|---|
| 登录身份/鉴权 | ✅ 0 影响（照用 `~/.codex` ChatGPT OAuth token） |
| 回复质量 | ✅ 0 影响（gpt-5.5、prompt 一字不改，只改通道不改内容） |
| 存量数据 | ✅ 0 影响（只动新轮） |
| 流式平滑度 | ⚠️ WS→HTTP SSE，吐字平滑度可能轻微体感差异，功能正常 |
| 单点风险 | ⚠️ 代理若挂→该轮重试 5 次失败约 25s→下轮自愈（35 分钟内未出现） |
| 部署依赖 | 需 `backports.zstd`（已装） |

### 回滚
opt-in 默认关。开 = 设 `CODEX_CAPTURE_PROXY_URL`+重启；**回滚 = 删该 env+重启**，瞬间回原生 WS 直连，零残留。

### bridge 重启姿势（坑，记一笔）
codex-bridge 是独立常驻 node 进程，由 transient 单元 `codex-bridge-3004.service` 管，读 `/run/user/1001/codex-bridge-3004.env`（由 event_service 从自己 `os.environ` 生成）。`restart.sh` **跳过 bridge 单元**，`_ensure_bridge` 见 bridge 活着就不重启。正确姿势=改 `.env` → **停 bridge** → 重启 event_service（懒加载，下条 codex 消息才拉起带新 env 的 bridge）。

## 5. 对后续训练的影响

### 5.1 代理手段本身 → 对训练零影响、零依赖
- **WS→HTTP 不污染数据**：只改传输通道，`/responses` 逻辑载荷逐字段一致；
- **采集与训练解耦**：代理只是采集期旁路组件，不进训练管线/最终模型；数据落盘后训练全离线，代理关掉/回滚不影响已收数据；训出的 qwen3 对 codex/代理/OpenAI 零运行时依赖。

### 5.2 codex 数据 → 一好两注意
- ✅ **格式占便宜**：codex 原生就是 OpenAI `/responses` 格式，离 canonical IR（OpenAI-messages 超集=verl 原生）最近，规范化变换最少，引错风险比 antigravity 残档低；
- ⚠️ **只能 BC、不能推理监督**：reasoning 永久加密 → 只能训 `输入→动作/文本`；
- ⚠️ **多轮夹加密块、训练前必剥**：深会话 input 会回传上一轮加密 reasoning 块（浅会话没有），必须剥（没法 tokenize、也是别的模型隐状态）。剥后副作用：部分动作 label 显得"欠上下文依据"，给 BC 注入轻噪声——但训 qwen3 本就不该塞 gpt-5.5 隐状态，可接受。

### 5.3 混源要统一输入格式（提前定）
codex（上下文无 CoT）和 claude（上下文带 thinking）混进同一 SFT 集 → 输入分布不一致。两条路：①**统一抹平**（claude 也剥 thinking，全集纯 BC，最稳）；②**显式标记**（给带 reasoning 的样本打标，让模型学"何时该想"，复杂度更高）。

## 8. codex SFT 归集（adapter 已落地，git `c722dd4`）

**adapter**：`trajectory-collect/scripts/codex_wire_adapter.py`。把 `codex_capture_proxy` 落盘的每条 wire `/responses` call 转成与 antigravity/claude 同构的归一化轨迹。

**与 antigravity 的本质区别**：
- antigravity：1 个 transcript = 1 会话事件流 → 1 条会话级记录（`per_call_boundary=False`）；
- **codex：1 个 `.jsonl.gz` = 多条独立 `/responses` call，每条本身就是 wire 逐字节** → 每条 call = 一个 `(input 上下文 → output 补全)` 样本，`completion_start_index` 给出 wire 真边界的 SFT 切分（`messages[:idx]`=输入，`messages[idx:]`=该 call 补全/target）。`per_call_boundary=True`。

**输出记录 schema**（顶层键）：
```
source="codex-wire", ts, path, status, model,
messages[],                 # system(base_instructions)+developer/user/assistant/tool
completion_start_index,     # 本轮补全在 messages[] 的起点（SFT target 边界）
tools[],                    # 官方 19 条完整 schema
usage,                      # {input_tokens, cached_tokens, output_tokens, reasoning_tokens, total}
sampling,                   # {temperature, top_p, reasoning_effort}
n_reasoning_stripped,       # 剥掉的加密 reasoning 块数
fidelity{...}               # 逐字段保真标注
```

**处理动作**：
1. **剥加密 reasoning 块**（`type=reasoning` 的 `summary=[]`+`encrypted_content` 不可读）→ 只留计数；
2. **保留官方 system/tools/usage/sampling**（codex 比 antigravity 多这几项）；
3. **脱敏**：复用 `secret-scrub` 单一事实源，`--publish` 强制过 + `chmod 600`，`--redact-pii` 把 open_id/app_id/内部 IP 伪匿名化（同值→同稳定代号）。

**1 条实测端到端验通**（2026-06-24）：管线 0 坏行、剥 1 个加密块、messages 重建正确（system+developer+user×2+assistant）、`completion_start_index=4` 切分干净、usage/sampling 保留、`--publish` PII 打码 6 处且无残留真实 open_id、加密块零残留。**完整 BC 集等 codex 自然多跑几轮后批量产出**（当前仅联调 1 条）。

**用法**：
```
python3 codex_wire_adapter.py --input ~/trajectory-data/codex-api-calls --out raw.jsonl       # 原始
python3 codex_wire_adapter.py --input ~/trajectory-data/codex-api-calls --publish pub.jsonl --redact-pii  # 脱敏发布
```
