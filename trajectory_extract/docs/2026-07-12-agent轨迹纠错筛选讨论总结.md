# 2026-07-12 聊天纠错筛选流程讨论总结

本文整理今天围绕“从聊天记录里筛出真实用户纠正”的讨论结论。重点不是导出 gold，也不是只补测试，而是确认当前流程能不能找到真实 C，以及重新筛选还要实际跑哪些步骤。

## 1. 当前目标

我们要从群聊事件流里抽出训练/评测用的纠错样本，结构是：

```text
t0 = 用户原始指令
B  = bot 说错的那句
C  = 真人触发的纠正
```

时序必须是 `t0 < B < C`。

这套流程要解决的核心问题：

- 找到 bot 的实质错误 B，而不是回执或系统提示。
- 找到真正触发纠正的真人消息 C，而不是后续 bot 道歉、自纠或系统压缩提示。
- 补齐 t0，形成完整 `t0 -> B -> C`。
- 判断 C 是否真的推翻 B，且是不是同一条线的问题。
- 生成可人工复核的核对台，再进入 gold/eval/replay。

## 2. v-final 判定口径

有效纠错样本必须同时满足：

- C 必须由真人触发；纯 bot 自纠不算。
- B 必须是 bot 的实质内容；系统提示、回执、审批、新指令都不能当 B/C。
- C 必须推翻 B；普通追问、照做、需求变更、新问题不算。
- “用户质疑触发返工”可以算 C，但返工修掉的错必须和用户质疑同线。
- 同一个 t0 下多个 B/C 要合并成一条 treatment。
- 单侧 session 里补不齐 `t0/B/C` 的候选要丢弃。

新增要特别锁住的一点：用户质疑触发返工还不够，返工修的错必须和用户质疑是同一条线，不能把顺手自纠的无关错误算进去。

## 3. 今天确认的事实错误样本

此前我把 `/opt/shared/data/task-trajectory/gold_review_sheet_pilot.md` 说成张耀明标注结果，或说它可以作为 #4 gold 来源，这是事实错误。

真人纠正消息是：

```text

```

这条本身应该作为标准纠错样本处理：

- t0：用户追问 #4 gold/标注来源可靠性。
- B：我把 `gold_review_sheet_pilot.md` 说成用户标注结果或可作为标注依据。
- C：真人质疑“文件我没标注过/你是不是搞错了”。

管线不能把后续 bot 承认错误、撤回文件、导出闸门或系统提示锚成 C。

## 4. 当前筛选流程

现在主流程不是一个脚本单跑到底，而是这条链路：

```text
runtime_lane.load_events()
  -> route_day
  -> session_corrections.py
  -> correction_census.validate_events
  -> verify_events
  -> user_corrections_pool*.jsonl
  -> recheck_user_corrections_pool.py
  -> verify_vfinal_recheck.py
  -> 核对台 / eval / replay
```

各环节职责：

| 环节 | 作用 |
|---|---|
| `runtime_lane.load_events()` | 读取聊天事件，保留 `msg_id / parent_id / role / name / text / ts`。 |
| `route_day` | 按回复链和模型把事件切成 session。 |
| `session_corrections.py` | 对每个 session 让模型枚举纠错候选，先产 B/C。 |
| `_resolve_anchor` | 用 `bot_error_quote` 回查真实 B，防止 B 锚错。 |
| `_resolve_corrector` | 如果 C 被锚到 bot/sys 且像自纠，回指到 B 和 bot 自纠之间的真人触发消息，优先看 `parent_id`。 |
| `validate_events` | 机器护栏：B 必须是 bot 实质消息，C 必须是真人，时序正确，quote 必须真在 B 里。 |
| `verify_events` | 对抗核验：判断 C 是否真的推翻 B，杀掉确认、无关追问、不同线返工。 |
| `export_session_corrections_pool.py` | 从 session 检测结果导出候选池。 |
| `recheck_user_corrections_pool.py` | 按 v-final 从窗口里重选 `t0/B/C`。 |
| `verify_vfinal_recheck.py` | 对 v-final accept 再跑核验。 |
| `build_gold_review_sheet.py` | 生成编号对话流核对台，供人工填写真值。 |
| `eval_gold_adjudications.py` | 对比 pipeline 输出和人工裁决，区分 recalled、只找到 B、C 锚错等情况。 |

## 7. 重新筛选还要做什么

接下来要做的是实跑重筛闭环：

1. 确认事件源里包含目标纠正消息，以及它的 `msg_id / parent_id / ts / role / name`。
2. 清理或备份旧 `session_corrections_raw.jsonl`，避免旧缓存绕过新 prompt 和回锚逻辑。
3. 重跑 `session_corrections.py`，生成新的 session 级候选。
4. 用 `export_session_corrections_pool.py` 导出候选池。
5. 跑 `recheck_user_corrections_pool.py`，按 v-final 补齐 `t0/B/C`。
6. 跑 `verify_vfinal_recheck.py`，杀掉 bot 自纠、系统提示、不同线追问。
7. 生成新的“对话流指认版”核对台。
8. 人工按 `真t0=L__ 真B=L__ 真C=L__` 裁决，并记录裁决来源 `msg_id`。
9. 跑 `eval_gold_adjudications.py`，检查目标样本是否从“B 找到但 C 锚错”变成 `recalled`。
10. 通过后再进入 replay/训练样本生成。

## 8. 核对台格式口径

核对台必须是“对话流指认版”，不再让人给预填 corrector 打勾。

要求：

- 每条消息编号为 `L1..Ln`。
- 标出作者类型：真人、bot、sys 忽略。
- 展示回复链，例如 `↩L24`。
- 管线猜测只作为旁注，例如 `«猜t0/B/C?»`。
- 只有真人行能当 C。
- 人工在卡尾填写 `真t0=L__ 真B=L__ 真C=L__`。

这样做的目的，是让人看到管线到底错在哪，而不是把错误的 corrector 继续包装成 gold。

## 9. 当前结论

当前流程已经从“旧候选池 + 错误 corrector 预填”推进到“session 级重筛 + C 回锚 + v-final 重核 + 对抗核验 + 对话流核对台”。

但最终 gold 仍不能来自 `gold_review_sheet_pilot.md`，也不能来自任何未带人工裁决来源的核对台。

合格输出必须满足：

- C 是真人消息。
- t0/B/C 都能在事件流中唯一定位。
- C 与 B 同线，且确实推翻 B。
- 有人工裁决来源 `msg_id` 后才能进入 gold。
- eval 能区分“只找到 B”和“B/C 都找对”。

下一步优先事项：拿 20260710 的 session-only 核对台做人工裁决，重点检查“文件我没标注过”这条是否被标成真实 C；然后跑 eval，确认 pipeline 是否正确 recalled。
