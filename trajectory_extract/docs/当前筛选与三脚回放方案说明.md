# 当前筛选与三脚回放方案说明

本文解释现在这套筛选/回放方案在做什么、为什么要这么做、字段怎么读，以及它哪里还不贴合“从错误轨迹里学习”的目标。

## 一句话版本

我们想验证一件事：把当年让 bot 犯错的用户输入重写得更清楚，并织入一条“不要再犯同类错”的抽象提示后，当前模型会不会更容易做对。

为了不自欺欺人，实验分三组：

| 组别 | 喂给模型的 user 输入 | 想回答的问题 |
|---|---|---|
| baseline | 原始用户输入 | 当前模型还会不会复现当年的错 |
| placebo | 只把原话说清楚，不加错误提示 | 是不是“写清楚一点”本身就够了 |
| treatment | 把原话说清楚，并加入抽象错误提示 | 错误轨迹里提炼出的提示是否真的有用 |

三组都用同一个 system：`你是一个严谨的 AI 助手。`  
也就是说，当前 QA 方案主要测“改 user 输入有没有用”，不是测“换 system prompt 有没有用”。

## 输入从哪里来

每条样本围绕一个三消息结构：

1. `t0`：用户最初交代任务的那句话。
2. `B`：bot 后来犯错的那句话。
3. `C`：真人用户指出、质疑或触发纠正的那句话。

例子：

```text
t0：帮我统计一下这批样本里有多少条合格。
B：我看完了，合格 42 条。
C：你根本没跑统计，怎么来的 42？
```

这条样本要学的不是“正确数字是多少”，而是“报数前必须真跑、不能凭感觉给数”。

## 当前筛选方案

当前筛选分两层。

第一层：找出疑似“用户纠正 bot”的样本。  
要求大致是：

- 纠正必须由真人触发，纯 bot 自己反省不算。
- 被纠正的 `B` 必须是 bot 的实质性错误陈述，不是“收到”“我去看看”这种回执。
- `C` 必须确实推翻 `B`，不能只是新需求、偏好变化或普通追问。
- 同一个 `t0` 后面多个错误，会合并到同一条 treatment 里，不重复算。

第二层：给回放实验准备“可注入的错误提示”。  
当前代码不直接把用户纠正原文塞给模型，而是从纠正轨迹里蒸馏一条更通用的准则，字段叫 `err_hint`。

例子：

```text
原始纠正：你根本没跑统计，怎么来的 42？
不该注入：这批不是 42 条。
可以注入：涉及数量、比例、统计结果时，必须先实际读取数据或运行统计；没跑就说明未验证，不能编数。
```

## `err_hint` 是什么

`err_hint` 是 treatment 重写 user 输入时使用的“错误抽象提示”。

它不是最终答案，也不应该包含当题的具体正确结果。它的作用是告诉重写器：当年这条输入诱发了哪类行为错误，重写时要把相应边界说清楚。

例子：

```text
原始用户输入：
看看这批 QA replay 结果怎么样。

err_hint：
报任何数字或结论前，必须先读取结果文件或运行统计；不能凭印象报数。

treatment 可能重写成：
请你读取 QA replay 的正式结果文件后再总结，不要凭记忆或估计报数。需要给出总条数、三脚各自通过情况和关键异常；如果某个数字没有亲手核对，就明确标成未验证。
```

## `错误提示来源` 是什么

`错误提示来源` 用来说明 `err_hint` 是怎么来的，以及有没有被防泄漏规则挡掉。

常见值：

| 来源文本 | 大白话解释 |
|---|---|
| `快照distilled-general(剔0条post_t0)` | 从快照里的通用准则拿到了可用提示；没有剔掉 t0 后证据 |
| `快照distilled-general(剔1条post_t0)` | 拿到了可用提示，但同时发现并剔掉了 1 条 t0 后的下游证据 |
| `过滤后无干净general(post_t0=1)→treatment退化placebo` | 唯一候选提示来自 t0 之后，当前规则不让用；treatment 没有纠错信号，只能退化成 placebo |
| `蒸馏准则(无快照)` | 没有完整快照，只能用旧结果里的蒸馏准则 |

## 当前“过滤”到底过滤什么

当前过滤是最保守的防泄漏策略。

如果某条纠正证据出现在 `t0` 之后，并且里面像是在给出具体失败、具体结论或裁决，代码会把它记到 `post_t0_failure_evidence`，不让它进入 `err_hint`。

这样做的目的：不要把答案倒灌回题目里。

例子：

```text
t0：帮我判断这段逻辑有没有 bug。
B：没有问题。
C：这里其实会空指针，应该先判断 user 是否为空。
```

当前保守规则会警惕 `C`，因为“会空指针”“先判断 user 是否为空”很可能已经是这道题的直接答案。直接塞进去，实验就变成偷看答案。

但这也带来一个问题：有些 t0 后信息虽然出现得晚，却不是答案，而是有价值的过程信号。

例子：

```text
C：你又没打开文件就下结论了。
```

这句话不告诉模型正确答案是什么，只指出失败模式是“没开盘核对就下结论”。按研究目标，这类信息其实应该能被抽象成安全提示。

## 三脚结果怎么判

每条样本会跑三组：

```text
baseline：原始输入
placebo：盲重写，只让话更清楚
treatment：带 err_hint 的重写
```

核心判定：

| 结果形态 | 怎么理解 |
|---|---|
| baseline 错，treatment 对，placebo 错 | 最理想：错误提示真的救回了 |
| baseline 错，treatment 对，placebo 也对 | 有改善，但可能只是“说清楚”救回了，不证明错误提示有效 |
| baseline 对 | 当前模型没有复现当年错误，这条没法证明“救回” |
| treatment 仍错 | 重写或提示没有压住问题 |
| 没有 `err_hint` | treatment 退化成 placebo，不计“纠错信号有效” |

例子：

```text
baseline：模型又报了没核对的数字 → 错
placebo：模型还是报了没核对的数字 → 错
treatment：模型说需要先读取结果文件，不能直接给数 → 对
```

这条就说明“报数前先核对”的提示可能有效。

再看一个混淆例子：

```text
baseline：模型硬编 → 错
placebo：只是把原话说详细后，模型就不硬编了 → 对
treatment：也对
```

这条不能说明 `err_hint` 有独立贡献，因为 placebo 也救回了。

## 当前方案的局限

第一，baseline 不是当年原环境复现。  
现在用的是当前模型、当前底座 system、压缩后的局部上下文。它能测“当前模型在这个构造场景下会不会错”，但不能严格还原“当年那个 bot 为什么错”。这也是 baseline 偏高的原因之一。

第二，当前过滤过保守。  
它把 `post_t0_failure_evidence` 整体挡在 `err_hint` 外面，能防止泄题，但也可能误杀“不含答案的失败模式”。

第三，QA 和 exec 的对照不一致。  
QA 现在有 baseline/placebo/treatment 三脚；exec 目前主要是 before/after A/B，还缺同等口径的 placebo。

第四，样本定位质量仍是上游关键风险。  
如果 `C` 锚错到 bot 自己的话、系统提示或普通回执，后面的 `err_hint` 和 replay 都会跟着偏。

## 建议下一版：加 `post_t0_safe_hint`

下一版不应简单地“t0 后全禁”或“t0 后全放”。

建议增加一个中间档：`post_t0_safe_hint`。

规则：

- 不能用直接答案、最终裁决、具体正确数字、具体修复代码、金标结论。
- 可以用不含答案的失败模式、检查策略、执行纪律。
- 每条 safe hint 都要能改写成通用准则，而不是这道题的答案。

例子：

```text
不能用：
这批 QA 是 11 条，baseline 对 9 条。

可以用：
汇总实验结果前必须读取正式结果文件；如果只是印象或临时日志，不能报成结论。
```

这样能更贴近研究目标：不是偷看答案，而是从 bot 犯错到被纠正这段轨迹里，学习“下次应该怎么检查、怎么表达边界、怎么避免同类失败”。

## 当前可执行结论

当前方案可以作为“最保守防泄漏版”继续跑，用来得到一组可信但偏保守的结果。

如果目标是验证“从错误轨迹里学习改进信号”，下一步应改造为：

1. 保留 baseline/placebo/treatment 三脚。
2. 保留当前 `err_hint` 的答案脱敏原则。
3. 增加 `post_t0_safe_hint`，允许 t0 后非答案型过程信号进入 treatment。
4. 给 exec 补 placebo，使 QA 和 exec 口径一致。
5. 先用人工小样本核 `post_t0_safe_hint` 是否泄题，再批量跑。
