## 一、要归集什么：RL 轨迹四要素

| 要素 | 来源 |
|---|---|
| 输入/Prompt（含动态 system prompt、工具 schema） | span `openclaw.content.system_prompt` / `input_messages` / `tool_definitions` |
| 模型输出/动作（含工具调用、CoT） | span `openclaw.content.output_messages` |
| 环境反馈（工具执行结果） | span `openclaw.content.tool_inputs` / `tool_outputs` |
| 奖励信号 | 暂为占位列：用户后续行为（评分/纠偏/采纳）按 session 回填 |

## 二、两条管道怎么选

**管道A：OTEL 被动归集（已落地）** —— 零侵入、文本级、天然支持延迟 reward。
适用：SFT / PRM（过程奖励模型）/ off-policy 训练。覆盖日常 80% 数据需求。

**管道B：API 劫持主动截获（按需再上）** —— slime 式 shim，改 `ANTHROPIC_BASE_URL`
指向 adapter，「字符串进、token 出」，推理时直接落 input_ids / output_ids / logprobs。
适用：真 on-policy PPO/GRPO——事后重分词有 token-drift 会污染梯度，文本级数据不够。
硬约束：推理引擎必须吐 token ID（vLLM ≥ 0.10.2；公网 Claude API、SGLang 不行），
所以只适用于自部署开源模型的训练循环。

## 三、管道A 落地架构（本机已跑通）

```
OpenClaw gateway（runtime 2026.6.5 + diagnostics-otel 插件）
   │  diagnostics.otel: traces=on, sampleRate=1.0, captureContent 六开关全开
   │  OTLP http/protobuf → 127.0.0.1:4318
   ▼
otlp_collector.py（systemd: trajectory-collector.service，纯 stdlib+protobuf）
   │  span 原样落盘 trajectory-data/spans/YYYY-MM-DD.jsonl
   ▼
trajectory_aggregate.py
   │  按 traceId 聚合 → trajectory-data/trajectories/<date>.jsonl
   ▼  每行一条轨迹：四要素 + token 计量 + reward 占位
```

代码：cbu-llm/skills `feishu-plugin/skills/trajectory-collect/`（SKILL.md 有运维命令）。

### 实施时踩的坑（重要，复现必看）

1. **属性名文档与实现不一致**：官方文档写 camelCase（`openclaw.content.systemPrompt`），
   插件实际导出 **snake_case**（`openclaw.content.system_prompt`）。聚合脚本按 snake_case 取。
2. **Node OTLP exporter 用 chunked 传输**：自写 collector 若只读 Content-Length 会拿到
   空 body（解出 0 span），且连接读坏后 exporter 停发。必须处理 `Transfer-Encoding: chunked`。
3. **runtime 版本门槛**：diagnostics-otel 插件要求 plugin API ≥2026.6.5，本机 openclaw
   已从 2026.6.1 升至 2026.6.5（npm 全局，分钟级可回滚）。
4. **span 非即时到达**：BatchSpanProcessor 分批 flush，触发后等 ~1 分钟再聚合。
5. **claude-cli provider 的轨迹形态**：模型调用发生在 Claude Code CLI 子进程内，
   gateway 侧主要拿到 tool.execution / run 级 span；模型正文捕获以 API 型 provider
   （openai-completions 等）最完整。

                 ## Tooling Available tools are policy-filtered...（完整拼装系统提示词 ✓）

                 请简单调用一个工具…','timestamp':1781268779050}, …]（含多轮历史 ✓）

验收结论：
- [x] 每条轨迹 trace_id 唯一、span 按时序可还原完整轮次
- [x] prompt（system+input）/ 模型输出 / 工具 schema 正文齐全（reward 占位）
- [x] 工具执行 span（tool.execution）随轮次落盘
- [ ] token 计量：`openclaw.model.usage` span 只在真实计费 provider 轮次出现
      （mock 轮无），聚合逻辑已就位，待 mimo 配额恢复后用真实流量复验

已知边界（如实记录）：
- **claude-cli provider 的轮次**只有 tool.execution / run 级 span，模型正文在 CLI
  子进程内、gateway 捕获不到——这类大脑走 Claude Code 内置 OTel 链路（见四·2，已落地）。
- `openclaw.diagnostic.phase` 等运维 span 会形成无正文的噪声 trace，聚合时用
  `--content-only` 过滤。

### 四·2 Claude Code 侧覆盖（2026-06-12 已落地，原「下一步②」）

`~/.claude/settings.json` env 块（单一修改点，bridge 子进程 / 手动 CLI / cron headless
全覆盖，随每次新会话生效、无需重启服务）开启内置 OTel，logs 汇入同一 collector。

**实测修正一个设计假设**：内置 OTel 并**不**导出模型输出正文——只有
`user_prompt`（正文，需 `OTEL_LOG_USER_PROMPTS=1`）、`api_request`（token/cache/cost
计量 + request_id）、`tool_decision` / `tool_result`（工具名/成败/字节数，无正文）。

正文补全方案：**OTel 提供计量与 join key（session.id），本地 transcript 提供正文**——
`~/.claude/projects/*/<session-id>.jsonl` 含完整 messages（assistant 输出、thinking、
tool_use/tool_result 全量）。`trajectory_aggregate.py --source claude-code` 按
session.id 聚合事件并自动 join transcript，输出 `trajectories/<date>.claude-code.jsonl`。

端到端验收（0612）：headless CLI 与 bridge 真实 bot 会话各一条，事件落盘 ✓、
transcript join ✓、正文齐全（含 thinking）✓、token/cost 计量与 SDK usage 一致 ✓。
