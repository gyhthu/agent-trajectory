# 聊天纠错筛选流程与重筛计划

日期：2026-07-12

## 结论

当前从聊天记录筛纠错的流程不是单脚本一次完成，而是一条分阶段链路：

`聊天事件源 -> session 切分 -> B/C 候选抽取 -> 机器护栏回锚 -> 对抗核验 -> 候选池 v-final 重核 -> 二次核验 -> 核对台/eval`

上一次提交 `9d800ce` 只完成了回归测试：证明“真人质疑文件/证据来源”这类纠正不能被锚到后续 bot 道歉或自纠。它还不是全量聊天记录重筛结果。

## 当前筛选流程

1. `runtime_lane.load_events()` 读取聊天事件。

   事件需要保留 `msg_id`、`parent_id`、`role`、`name`、`text`、`ts` 等字段，后续所有 t0/B/C 定位都依赖这些原始字段。

2. `route_day` 按回复链和模型上下文切 session。

   这一步把连续聊天拆成可分析窗口，避免模型在全群长上下文里乱跨话题。

3. `session_corrections.py` 从每个 session 中抽取候选 B/C。

   B 是 bot 说错的实质内容，C 是真人触发的纠正。脚本先产候选字段，例如 `anchor_msg_id` 指向 B，`corrector_msg_id` 指向 C。

4. `correction_census.validate_events` 做机器护栏。

   主要检查：

   - B 必须是 bot 的实质消息，不能是回执、系统提示或空泛状态。
   - C 必须是真人消息，且晚于 B。
   - 如果模型把 C 锚到 bot 后续“我错了/我撤回”这类自纠，护栏要回指到 B 和 bot 自纠之间真正触发返工的真人消息。
   - 用 `bot_error_quote` 反查 B 原文，防止 B 锚错。
   - 排除系统提示、ack、纯需求变更、不同线追问等非 v-final 样本。

5. `verify_events` 做对抗核验。

   这一步重新喂上下文窗口，让模型判断 C 是否确实推翻 B，尤其要区分“用户同线质疑触发返工”和“bot 顺手自纠了别的问题”。

6. `user_corrections_pool.jsonl` 汇总候选池。

   候选池承接多条来源：当前 session 抽取、历史 guard/verify 存活项，以及此前误杀捞回池。

7. `recheck_user_corrections_pool.py` 按 v-final 重核。

   这一步要求补齐三消息模型：t0 是用户原指令，B 是 bot 错误，C 是真人纠正。不能把 C 当 t0，也不能用 bot 自己的承认错误充当 C。

8. `verify_vfinal_recheck.py` 二次核验。

   二次核验继续杀掉 bot 自纠、系统提示、不同线追问、纯设计偏好和需求变更。

9. 生成核对台或进入 eval。

   核对台只应列能唯一定位的候选；不能唯一定位的样本应进入缺口清单。gold 导出现在还要求人工裁决来源 `msg_id`，避免把二手材料误当人工标注结果。

## 这次暴露的问题

真实问题不是“缺一个测试”，而是全量重筛还没跑。

已经补上的测试覆盖了一个关键回归：当模型找到 B，但把后续 bot 道歉、自认错误或撤回说明当成 C 时，系统必须判错或回锚到前面的真人质疑。

但测试通过只说明规则被锁住，不等于历史聊天记录已经按新规则重新筛过。

## 重新筛选还要做什么

1. 确认真实聊天事件源包含“文件/证据来源被质疑”的那条真人消息，并核对它的 `msg_id`、`parent_id`、时间戳和作者类型。

2. 重新跑 `session_corrections.py`。

   必要时清理旧的 `session_corrections_raw.jsonl` 缓存，防止旧模型输出绕过新的 prompt 和 C 回锚规则。

3. 重建或更新 `user_corrections_pool.jsonl`。

   目标是让新筛出的真人 C 进入候选池，而不是停留在测试样本里。

4. 跑 `recheck_user_corrections_pool.py`。

   对候选池逐条补齐 t0/B/C，并按 v-final 定义重新解释。

5. 跑 `verify_vfinal_recheck.py`。

   继续过滤 bot 自纠、系统提示、不同线返工和无法唯一定位的候选。

6. 生成新的核对台。

   新核对台应采用对话流指认格式：编号行 `L1..Ln`、标明真人/bot/sys、展示 `parent_id` 回复链，并让人工只从真人行里选 C。

7. 跑 eval。

   重点看这条“B 找到了但 C 锚错”的样本是否从错误类型变成 recalled。如果仍找不出真实 C，再改主链路，而不是只补测试。

## 若重筛仍失败，主链路要改的规则

1. C 只认真人消息。
2. C 定位优先走 `parent_id` 回复链。
3. 防止把纠正句本身错当 t0。
4. 判据 3(c) 必须同线：用户质疑触发返工还不够，返工所改错误必须与用户质疑是同一条线。
5. 系统提示、压缩提示、ack、审批口令和 bot 自我道歉不能当 B/C。
